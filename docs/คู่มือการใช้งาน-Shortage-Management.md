## 🎯 ภาพรวมระบบ

### ระบบทำอะไรได้บ้าง?

✅ **ตรวจจับสินค้าขาดอัตโนมัติ** - ระบบจะตรวจสอบ 2 จุด:
- **จุดที่ 1**: ตอนสร้าง Batch (PRE-PICK) → ถ้าสต็อกไม่พอ
- **จุดที่ 2**: ตอนหยิบของ (POST-PICK) → ถ้าหาของไม่เจอ/ของชำรุด

✅ **บันทึกสาเหตุที่ชัดเจน** - มี 6 สาเหตุ:
1. 🔍 หาไม่เจอ (ปัญหา Layout/การฝึกอบรม)
2. 💔 ของชำรุด (ปัญหา QC)
3. 📦 วางผิดที่ (ปัญหา Process)
4. 🏷️ บาร์โค้ดหลุด (ปัญหา Material)
5. ❌ ไม่มีในระบบ (ปัญหา Inventory)
6. 📝 อื่นๆ (สาเหตุอื่นๆ)

✅ **วิเคราะห์รากเหง้าปัญหา** - Dashboard แสดง:
- สินค้าขาดแบบไหนมากที่สุด (PRE-PICK vs POST-PICK)
- สาเหตุไหนเกิดบ่อย
- SKU ไหนมีปัญหาบ่อย
- Trend การเกิด shortage

✅ **เก็บบันทึกทุกการเปลี่ยนแปลง** - Transaction Log:
- เมื่อไรเกิด shortage
- ใครทำ shortage
- จัดการยังไง (รอสต็อก/ยกเลิก/แทน SKU)
- ใครแก้ไข

**💡 Tips สำหรับ Picker**:

✅ **DO - ควรทำ**:
- ✅ เลือกสาเหตุที่ตรงความจริง
- ✅ ใส่หมายเหตุถ้ามีรายละเอียดเพิ่มเติม
- ✅ แจ้งหัวหน้างานถ้าเจอปัญหาซ้ำๆ

❌ **DON'T - ไม่ควรทำ**:
- ❌ เลือก "อื่นๆ" ทุกครั้ง (ไม่ช่วยวิเคราะห์)
- ❌ Skip การใส่หมายเหตุ (ถ้ามีข้อมูลเพิ่มเติม)
- ❌ หยิบไม่ครบแต่ไม่ Mark Shortage
- ❌ Mark Shortage ก่อนค้นหาจริงจัง

**ตัวอย่างการเลือกสาเหตุที่ถูกต้อง**:

| สถานการ์ | สาเหตุที่ควรเลือก | เพราะอะไร |
|---------|------------------|----------|
| ไปที่ Loc. A-01-05 แต่ไม่มีของ | 🔍 หาไม่เจอ | ของไม่อยู่ตำแหน่งที่ระบุ |
| เจอของแต่กล่องแตก/เสียหาย | 💔 ของชำรุด | ของมีแต่ใช้งานไม่ได้ |
| เจอของที่ Loc. อื่นไม่ใช่ Loc. ที่ระบุ | 📦 วางผิดที่ | ของถูกวางผิดตำแหน่ง |
| สติกเกอร์บาร์โค้ดหลุด/ขาดหาย | 🏷️ บาร์โค้ดหลุด | มีของแต่เช็คไม่ได้ว่าใช่หรือเปล่า |
| ระบบบอก Loc. แต่ไปดูไม่มีของเลย | ❌ ไม่มีในระบบ | ข้อมูลผิดพลาด |

### 🔧 สำหรับ: หัวหน้าคลัง / Supervisor

#### 3️⃣ จัดการ Shortage Queue

**ไปที่**: `/shortage-queue` (หน้า Shortage Queue)

**สิ่งที่จะเห็น**:

```
┌────────────────────────────────────────────────────────────────┐
│  Shortage Queue                                    [ + Filter ]│
├────┬──────────┬─────┬────────┬─────────┬──────────┬───────────┤
│ ID │ Order ID │ SKU │ ประเภท │ เหตุผล   │ หมายเหตุ  │ Action   │
├────┼──────────┼─────┼────────┼─────────┼──────────┼───────────┤
│ 45 │ TH001234 │ ABC │PRE-PICK│❌ไม่มี  │สต็อกไม่พอ│[Action ▼]│
│ 44 │ TH001235 │ XYZ │POST-PICK│🔍หาไม่เจอ│ชั้น B-02 │[Action ▼]│
│ 43 │ TH001236 │ DEF │POST-PICK│💔ชำรุด  │กล่องแตก  │[Action ▼]│
└────┴──────────┴─────┴────────┴─────────┴──────────┴───────────┘

📊 สถิติ:
   • รอดำเนินการ: 12 รายการ
   • รอสต็อก: 8 รายการ
   • จัดการแล้ว: 25 รายการ
```

**Actions ที่ทำได้**:

1️⃣ **รอสต็อกเข้า** (Wait Stock)
```
เมื่อใช้: สินค้ากำลังสั่งเข้า/กำลังมาถึง

ขั้นตอน:
1. คลิก Action → เลือก "รอสต็อกเข้า"
2. ใส่หมายเหตุ (optional): "รอ PO-12345"
3. กด "บันทึก"

ผลลัพธ์:
✅ Status → waiting_stock
✅ Order ยังคงอยู่ในระบบ
✅ เมื่อสต็อกเข้า → กลับไปจัดการต่อ
```

2️⃣ **ยกเลิก** (Cancel)
```
เมื่อใช้: ลูกค้ายกเลิก / ไม่มีของแทน

ขั้นตอน:
1. คลิก Action → เลือก "ยกเลิก"
2. ใส่หมายเหตุ: "ลูกค้ายกเลิก order"
3. กด "บันทึก"

ผลลัพธ์:
✅ Status → cancelled
✅ Release stock reservation
✅ บันทึก transaction log
❌ Order นี้จะไม่จัดส่ง
```

3️⃣ **แทน SKU อื่น** (Replace SKU)
```
เมื่อใช้: มี SKU อื่นใช้แทนได้

ขั้นตอน:
1. คลิก Action → เลือก "แทน SKU อื่น"
2. กรอก SKU ใหม่: "XYZ-789"
3. ใส่หมายเหตุ: "ใช้ XYZ-789 แทน ABC-123"
4. กด "บันทึก"

ผลลัพธ์:
✅ Status → replaced
✅ บันทึก replacement_sku
✅ ส่ง SKU ใหม่แทน
```

4️⃣ **จัดการเรียบร้อย** (Resolved)

## ❓ FAQ คำถามที่พบบ่อย

### 🤔 Q1: PRE-PICK กับ POST-PICK ต่างกันยังไง?

**A:** แตกต่างกันตรง **จุดที่เกิด** และ **สาเหตุ**

| หัวข้อ | PRE-PICK | POST-PICK |
|--------|----------|-----------|
| **เกิดเมื่อไร** | ตอนสร้าง Batch | ตอนหยิบของ |
| **ใครพบ** | ระบบตรวจสอบอัตโนมัติ | Picker พบขณะทำงาน |
| **สาเหตุ** | สต็อกในระบบไม่พอ | ของไม่อยู่/หาไม่เจอ/ชำรุด |
| **หน้าที่แก้** | ทีมจัดซื้อ/Inventory | ทีมคลัง/Operations |
| **ตัวอย่าง** | Order ต้องการ 10 แต่มี 5 | Order ต้องการ 10 ระบบบอกมี 10 แต่หาไม่เจอ |

### 🤔 Q2: ทำไมต้องเลือกสาเหตุ? เลือก "อื่นๆ" ทุกครั้งไม่ได้เหรอ?

✅ **ประโยชน์ของการเลือกสาเหตุที่ถูก**:
- Manager รู้จุดปรับปรุง
- ทีมแก้ปัญหาได้ตรงจุด

```
ถ้าทุกคนเลือก "อื่นๆ":

```

### 🤔 Q3: "Resolved" กับ "Cancel" ต่างกันอย่างไร?

**A:**

| Action | Resolved | Cancel |
|--------|----------|--------|
| **ความหมาย** | จัดการเรียบร้อยแล้ว | ยกเลิก Order |
| **ใช้เมื่อ** | หาของเจอแล้ว/หยิบครบแล้ว | ลูกค้ายกเลิก/ไม่มีของแทน |
| **Order** | ส่งต่อได้ | ไม่ส่ง |
| **เงื่อนไข** | ต้องหยิบของให้ครบก่อน | ไม่ต้องหยิบ |

**ตัวอย่าง Resolved**:
```
สถานการณ์: Picker หาไม่เจอ → Supervisor ไปหา → เจอแล้ว
→ Supervisor หยิบของให้ครบ
→ Action: "Resolved"
→ Order ส่งได้ ✅
```

**ตัวอย่าง Cancel**:
```
สถานการณ์: หาของไม่เจอจริงๆ + ไม่มี SKU ทดแทน + ลูกค้าบอกยกเลิก
→ Action: "Cancel"
→ Order ไม่ส่ง ❌
→ คืน Stock Reservation
```

3. **เลือกสาเหตุที่ถูก** = ช่วยวิเคราะห์ปัญหา
4. **ใส่หมายเหตุ** = ช่วยทีมเข้าใจบริบท
5. **Resolved ต้องหยิบครบก่อน** = ไม่งั้น Error

### 📊 การใช้งานประจำวัน

**Picker**:
- Scan → หยิบ → ถ้าไม่ได้ → Mark Shortage
- เลือกสาเหตุที่ถูก
- ใส่หมายเหตุถ้ามี

**Supervisor**:
- เช็ค Shortage Queue ทุกวัน
- จัดการ pending shortage
- ติดตาม waiting_stock

**Manager**:
- ดู Analytics ทุกสัปดาห์
- วิเคราะห์ Top SKUs
- ปรับปรุง process
